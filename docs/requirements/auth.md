# 認証・招待・組織所属

## 概要

本システムは、ユーザー認証および組織単位のアクセス制御を行う。  
ユーザー（user）は組織（organization）に所属し、役割（role）に応じて権限が異なる。

## 本書の範囲

- 認証・招待・組織所属に関する**要件定義**（満たすべき挙動）を記載する。
- 画面遷移・ボタン・入力項目の詳細は `docs/ui` に記載する。
- ロールごとの操作可否の網羅は `docs/permissions` に記載する。
- エンドポイント・リクエスト形式・レスポンス形式は `docs/api` に記載する。
- テーブル定義・インデックス・マイグレーションは `docs/database` に記載する。
- アクセストークンのクレーム構成・organization コンテキストの載せ方は `docs/api` および `docs/architecture` に記載する。
- ログイン後に利用するドメイン機能（タスク等）の要件は `docs/requirements/tasks.md` 等の各要件書を参照する。

## 用語定義

- User: ユーザー（ログインする個人）
- Organization: 組織（会社・チーム）単位
- Membership: ユーザーと組織の関係
- Role: 役割（ユーザーの権限区分）
- Invite: 招待情報

## 機能要件

### ログイン

- **メールアドレス**（email）と**パスワード**（password）による認証。
- ログイン成功後は**セッションベース**または**トークンベース**（Bearer トークン等。例として JWT も可）とし、形式・有効期限・失効・複数端末は `docs/api`（必要に応じて `docs/architecture`）で定義する。
- **アクセストークン**（または同等の認可情報）に **organization コンテキスト**（少なくとも現在の操作対象 `organization_id`）を含める。クレーム名・更新タイミング・切替 API は `docs/api` および `docs/architecture` で定義する。
- **セッション方式**を採用する場合は、**サーバー側セッション**（または同等のサーバー主導ストア）に現在の `organization_id` を保持し、リクエストごとに**トークン方式と同様**に organization を解決できること。クッキーに載せる場合の属性（`HttpOnly` 等）は `docs/api` / `docs/architecture` で定義する。
- **ログアウト**時は、セッション無効化またはアクセストークン失効を行う。具体的手順・エンドポイントは `docs/api` に記載する。
- 認証失敗時のレート制限・一時ロックは**非機能要件 > セキュリティ**を参照する。利用者向けエラー表示の文言は `docs/ui` で定義する。

### 新規登録（管理者）

- **組織単位**の新規作成
- ユーザーは**管理者**として登録される。

### メール認証

- 管理者の新規登録時はメール認証を必須とする。
- 登録直後はメール未認証（`email_verified_at = null`）として作成する。
- `email_verified_at` は、**当該 email をユーザーがメール経路で証明した時点**の日時とする。手段として (a) 管理者向けメール認証リンクの完了、(b) 招待の受諾に伴う**新規 user の作成完了**を認める。いずれも同一カラムで「メール検証済み」を表す。
- `email_verified_at` は、確認手段の違いを**区別しない**（手段別のカラムや列挙値は設けない）。
- 管理者については、認証メール内のリンクを開いた時点で (a) に該当し、`email_verified_at` に日時を保存する。
- 招待で**新規に user を作成する**場合は、ワンタイム招待リンクと招待先メール一致により (b) に該当し、**受諾成功時**に `email_verified_at` に日時を保存する。管理者向けメール認証リンクによる**追加のメール認証フローは行わない**。
- **既存 user** が招待を受諾する場合はユーザーの新規作成を行わないため (b) は該当せず、`email_verified_at` は**変更しない**。
- 未認証ユーザー（`email_verified_at = null`）は**ログインは可能**とするが、**organization に紐づく操作は一切行えない**。
- 認証リンクの期限・トークン・再送・冪等性は**非機能要件**（**トークン仕様（共通）** / **可用性・運用性**）を参照する。
- 招待リンクの期限切れ・使用済み時は**再招待**する。

### パスワードリセット

- 利用者が **email** を入力してリセット用メールを送信する。入力値は `users.email` と同じ正規化（小文字化・前後トリム）を適用してから照合する。
- メール内リンクに**平文トークン**を含め、クリック後に新パスワードを設定できるようにする。
- アカウント列挙対策は**非機能要件 > 可用性・運用性**を参照する。
- トークン仕様（生成・保存・有効期限・失効）および無効トークン時の取り扱いは**非機能要件 > トークン仕様（共通）**を参照する。

### 招待（発行・受諾）

#### 管理者による発行

- 管理者がメールアドレスを指定して招待する。
- 招待トークンを発行する。
- 同一組織・同一メールに対し、**有効な未使用**の招待が既にある場合は新規作成せず、**再送**を優先する。

#### 被招待者の登録

- 招待リンク（トークン）経由で登録する。
- `email_verified_at` は、新規 user 作成の受諾成功時に設定し、既存 user の受諾では変更しない（「メール認証」と同一方針）。
- 指定された **organization** と **organization ロール**（`memberships.role`）で参加する。プロジェクト単位の権限は、参加後に `project_memberships` で別途付与する。
- 招待先メールアドレスと登録メールアドレスを一致させる（正規化後の値で一致させる）。
- 既存ユーザーが同一メールで招待を受諾する場合は、ユーザーを新規作成せず既存 user を organization に紐付ける。

#### フロー

1. invite 作成
2. メール送信
3. ユーザー登録（または既存ユーザーの紐付け）
4. organization に紐付け
5. 招待トークンを無効化（ワンタイム。`used_at` を設定する等）

#### invites の一意性（方針）

- **`(email, organization_id, revoked_at IS NULL, used_at IS NULL)` の組み合わせで一意性を担保する**（部分一意インデックス等）。正規化後の `email` で比較する。これにより、**有効な未使用**の招待は同一組織・同一メールで同時に高々1件となる。

#### invites の状態

次の区分は**排他的**に扱う（同一レコードはいずれか一つの状態にのみ分類する）。判定は **UTC** の現在時刻で行う。

1. **取り消し済み**: `revoked_at IS NOT NULL`
2. **使用済み**: `revoked_at IS NULL` かつ `used_at IS NOT NULL`
3. **期限切れ**: `revoked_at IS NULL` かつ `used_at IS NULL` かつ `expires_at < 現在時刻`
4. **未使用（有効期限内）**: `revoked_at IS NULL` かつ `used_at IS NULL` かつ `expires_at >= 現在時刻`

上記の優先順は **1 → 2 → 3 → 4** とする（DB 上は複数条件が同時に真になり得る列の組み合わせでも、運用上の状態は一意に決める）。

#### エラー・例外時の扱い

- 無効・改ざん・期限切れ・使用済みトークンは受け付けず、**要求を拒否**する。利用者向けの表示・文言は `docs/ui` で定義する。
- 受諾時に招待先メールアドレスと一致しない場合は登録不可とする。
- 組織から招待が取り消された場合はリンクを無効化する。
- 既に同一 organization へ参加済みの場合は受諾を完了扱いとし、重複 membership は作成しない。

### エラー・例外時の扱い（共通）

- 存在しないリソース（ユーザー・組織・招待等）を指定した場合は要求を拒否する。
- 認証情報が不正または欠落している場合は要求を拒否する（未認証として扱う）。
- メール未検証・organization 未所属・organization コンテキスト未選択のいずれかに該当する場合の拒否条件は、**メール認証**および**組織コンテキスト**の要件に従う。
- 認証済みであっても、操作対象に対する権限がない場合は要求を拒否する。
- メール認証・パスワードリセット・招待における無効・改ざん・期限切れ・使用済みトークンは、各機能要件および**非機能要件 > トークン仕様（共通）**に従い拒否する。
- ユーザー自身のプロフィール更新や組織設定更新など、**更新の競合制御**（楽観ロックの有無・条件、不一致時の応答）は `docs/api` で定義する。

## 役割・権限・認可

役割によって権限を付与し、操作を制御する。細目は `docs/permissions` を参照する。

- **organization ロール**は `memberships.role`、**プロジェクト単位の権限**は `project_memberships` で管理する。
- タスク等の操作可否は上記の組み合わせで決定する。タスクの細目は `docs/requirements/tasks.md` と `docs/permissions` を参照する。

### 管理者（admin）

- 全メンバーの管理
- 全データの操作可能

### プロジェクトリーダー（project_leader）

- プロジェクト参加メンバーの管理
- 各プロジェクトに関するデータの操作可能

### メンバー（member）

- プロジェクトタスクの操作
- 一部編集可能

### 閲覧者（viewer）

- 閲覧のみ

## 組織コンテキスト

- ユーザーは複数の organization に所属可能とする。
- ユーザーが organization に1件も所属していない場合、**organization に紐づくすべての機能（タスク・プロジェクト等）を利用できない**。
- ログイン後は利用する organization を決める（既定 organization の有無・複数所属時の切替は、**アクセストークンに含める organization コンテキスト**およびサーバー側セッション等で整合させる。詳細は `docs/api` を参照）。
- organization コンテキストが未選択の状態では、**organization に依存するすべての操作を禁止する**。
- **organization に依存するすべての API** は、**organization コンテキストの一致検証**（トークンに含まれる `organization_id` と、リクエストで解決される organization が一致すること）を**必須**とする。検証ルール・エラーコードは `docs/api` で定義する。
- 複数所属時は、organization を選ぶ手段を提供する（UI の詳細は `docs/ui`）。

### organization からのメンバー除外（削除）

- ユーザーが organization から削除された場合、当該 `(user_id, organization_id)` の **`memberships` は削除する**。
- 当該ユーザーに紐づく業務データ（タスク等）の扱いは、**各ドメイン要件**（例: `docs/requirements/tasks.md`）で定義する。

## 非機能要件

### トークン仕様（共通）

- 招待・メール認証・パスワードリセットで用いる秘密トークンは、暗号学的乱数で生成する（目安: 32バイト / 256bit 相当以上）。
- トークンの URL 表現は Base64URL 等を用いる。
- DB には平文トークンを保存せず、一方向ハッシュのみを保存する。検証時は受信値を同じ方式でハッシュして照合する。
- トークンは用途ごとに有効期限を設け、期限切れ時は無効として扱う。
- トークンはワンタイム利用を原則とし、再送・再発行時は未使用の旧トークンを失効させる。

### セキュリティ

- 通信は HTTPS を前提とする。
- パスワードは 8文字以上とし、保存時はハッシュ化する（bcrypt 等）。
- ログイン試行、メール認証再送、パスワードリセット依頼、招待再送にはレート制限を設ける。
- 連続ログイン失敗に対する一時ロック（例: 5回失敗で15分）を適用する。
- トークン値そのものを監査ログやアプリケーションログに出力しない。

### 可用性・運用性

- パスワードリセット時はアカウント列挙対策として、登録有無にかかわらず同一の応答メッセージを返す。
- 認証済みリンクや重複受諾などの再実行は、可能な範囲で冪等に成功として扱う。
- トークン期限（招待/メール認証: 24時間、パスワードリセット: 60分）は設定値で変更可能とする。
- 時刻判定はサーバー時刻（UTC）で統一し、表示時のみローカルタイムゾーンへ変換する。
- 認証・セッション・トークン関連 API の応答時間目標やタイムアウトは `docs/api` で定義する。

### 監査可能性

- 主要イベント（登録、認証、リセット、招待作成/再送/受諾/失効）を監査ログに記録する。
- 監査ログには実行者、対象、組織、結果、実行時刻を記録し、追跡可能性を担保する。
- 記録項目の粒度・保存期間・参照権限は `docs/permissions` および運用方針に合わせ `docs/api` で必要に応じて補足する。

## 備考（運用ルール）

- メール送信（テンプレート、送信元、再送 UI）は `docs/ui` およびメール基盤の設計書で定義する。
